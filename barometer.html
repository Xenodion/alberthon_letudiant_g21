Build vs Partner
Framework de décision · Lead Scoring Model · logique argumentée
Question L'Étudiant : Pour le lead scoring model, faut-il construire en interne ou s'appuyer sur un partenaire d'enrichissement (Cognism, Clearbit, Lusha…) — et quelle est la logique de décision ?
Option A
Build In-house
+Données comportementales letudiant.fr exclusives : non disponibles chez des tiers
+Contrôle total du modèle : adaptation aux cycles saisonniers (rentrée, Parcoursup, salons)
+Conformité RGPD native : données dans le périmètre légal de L'Étudiant
+Pas de coût de licence tiers : marge 85–93% préservée
+Actif stratégique : le modèle devient une barrière à l'entrée long terme
−Nécessite une compétence data science interne ou une embauche
−Time-to-market plus long : 3–6 mois pour un modèle ML en production
★ Recommandée · Option B
Hybride (Build core + Partner enrichissement)
+Modèle core in-house sur données propriétaires · différenciateur préservé
+Enrichissement géographique INSEE open data (gratuit, sans DPA complexe) pour les 3% sans région
+Partenaire commercial (Cognism, Clearbit) uniquement si Life-moment API lancée et clients B2B demandent signaux cross-domain
+Couplage lâche : remplacement du partenaire facilité, pas de lock-in
−Coût de licence si partenaire commercial : 500–3 000 €/mois selon volume
−DPA requis avec tout tiers : due diligence RGPD à prévoir
Framework de décision par critère
Logique appliquée critère par critère
CritèreDécisionRaisonnement
Unicité des données comportementalesBUILDSignaux chatbot, simulateur, salons sur letudiant.fr exclusifs. Cœur différenciateur non reproductible par un tiers.
Signaux complémentaires (géo, socio)PARTNERRégion NULL sur ~3% des profils (97% couverts grâce à la jointure BigQuery). INSEE open data suffit pour combler sans DPA complexe.
Logique de scoring compositeBUILDModèle fraîcheur/intention/complétude codé, testé, documenté. Le reconstruire chez un tiers serait régressif.
Time-to-market initialBUILDPipeline opérationnel aujourd'hui. Pas besoin de tiers pour le MVP et le pilote commercial.
Conformité RGPDBUILDToute intégration tiers nécessite un DPIA. Pour le MVP, rester dans le périmètre BigQuery de L'Étudiant est la voie la plus simple.
Scalabilité long terme (Y2+)HYBRIDEBigQuery ML ou Vertex AI pour le modèle core. Partenaire commercial optionnel si Life-moment API décollage et clients B2B demandent des signaux externes.
Recommandation finale

Architecture hybride en deux temps : (1) Lancer avec le pipeline in-house actuel sur BigQuery — c'est suffisant pour le MVP et le pilote. (2) À partir de la production, évaluer un enrichissement géographique INSEE (gratuit) pour les profils sans région. Les fournisseurs commerciaux ne sont pertinents qu'en Y2 si des clients Life-moment API demandent explicitement des signaux cross-domain — décision à revoir à ce moment-là.
